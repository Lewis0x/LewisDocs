# 需求文档

## 1. 背景与动机

研究团队在 2024-2026 年期间累积了一份名为「通用 CAD 平台 API 设计哲学」的研究系列：

- **总量**：11 份 Markdown 文档，约 17.4 万字
- **结构**：三层金字塔（导读 1 篇 / 理论顶层 1 篇 / 横向对比 1 篇 / 厂商深度剖析 8 篇）
- **特点**：43 处 `[回链：3.x §一/§二 …]` 形式的纯文本交叉引用

原始材料是分散的 `.md` 文件，存在两个核心痛点：

1. **不可点击的交叉引用** — 43 处回链全部是纯文本，读者必须人肉切换文件 + 滚动定位
2. **无法跨文档搜索** — 11 份文档需逐个 `Ctrl+F`，做不到「几何内核」这类术语的全局检索

第二层痛点是分发：

3. **本地分发繁琐** — 邮件传 ZIP / 共享盘，更新一次要全员同步
4. **PDF 失去交叉跳转** — 即使打成 PDF，跨文档跳转仍受 PDF Reader 限制

## 5. 范围边界（明确不做）

为了控制复杂度并避免「研究站点变成产品」的范围蔓延，以下事项**显式不做**：

- ⚠️ 不改写 V4 源文档的**论述内容**（判断、推论、证据、表格、文字）
- ✅ **允许**升级 V4 源文档的**呈现形式**：把核心 ASCII 框图替换为 Mermaid（仅结构改写、不改语义；改写需在 02-design.md 的 ADR 中留痕）
- ❌ 不做多语言（i18n）—— 内容只有中文版
- ❌ 不做版本切换 —— 文档治理已收敛到 V4 单版本
- ❌ 不接付费搜索服务（Algolia DocSearch 等）
- ❌ 不自定义复杂主题 —— VitePress 默认主题已够用
- ❌ 不做评论 / 标注 / 用户系统
- ❌ 不做 PDF 导出 —— 浏览器自带打印即可
- ❌ 不做访问统计 —— 反爬已加（F12），但用户行为统计仍按需
- ❌ 不做对用户可见的反爬元素 —— 不加 footer 版权声明、不禁右键、不上登录墙
- ❌ 不接 hCaptcha / reCAPTCHA —— 仅用 Cloudflare Turnstile 隐式挑战

## 7. 关键约束

- **源文档单向流动** —— `source/` → `docs/`，永远不反向。如发现内容错误，必须改 V4 上游再重跑 `prepare-content`。
- **路由稳定** —— 一旦 URL 形成（如 `/platforms/onshape#三-…`），不再随意改文件名或章节标题，否则破坏外部书签。
- **构建可重复** —— 任意时刻在干净环境跑 `npm ci && npm run prepare-content && npm run build`，产物字节级一致（除时间戳外）。

| V3（不计划） | 接入对外读者 | 视情况加访问控制 |

> 任何超出 V2 的改动需先回头改本需求文档，再进入实施。
