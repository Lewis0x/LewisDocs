## 一、系列概览

### 1.1 这是关于什么的研究？

这是一份**关于通用 CAD 平台 API 设计哲学的系统研究**——不绑定具体厂商立场，从 8 个主流 CAD 平台（AutoCAD、CATIA、NX、Onshape、MicroStation+iTwin、SolidWorks、SketchUp、FreeCAD）的 API 设计中**提取共性模式、差异哲学、演进规律**，并归纳为可供新平台架构师参考的思考框架。

⚠️ **本系列的定位边界**：研究素材偏向参数化 CAD（5/8 样本是参数化），所有论断应理解为**对样本平台的归纳**，而非对所有 CAD 平台的普遍规律。本系列采用"思考工具"而非"正确答案"的姿态。

## 三、核心要点速查

### 3.1 文档 1 的几条主论断

| 论断 | 证据等级 |
|---|---|
| CAD API 反映商业战略 | A |
| 长期兼容承诺与平台权威相关 | B |
| 四大支柱的选择互相约束（在样本中） | B |
| 演进趋势存在但有边界（声明化在参数化 CAD 中明显，云原生不是终点等） | B / C |

⚠️ **关于证据等级**：A/B/C 是作者**写作辅助标签**，**不是客观评分**。仅用于提示读者哪些论断有更扎实的样本支撑、哪些是作者归纳。

### 3.2 四大支柱

```
在本文样本覆盖的平台中，API 均可映射到四类支柱：

支柱 1：几何（B-Rep / NURBS / Mesh / Face-based）
支柱 2：对象数据（File-based vs Database-based、关联性传播）
支柱 3：扩展数据（6 级 schema 光谱，强度匹配数据集成需求）
支柱 4：事件与协作（颗粒度光谱、协作模式 6 阶段）
```

### 3.4 8 个平台一句话定位

| # | 平台 | 一句话定位 |
|---|---|---|
| 3.1 | **AutoCAD ObjectARX** | 30+ 年 in-process 原生 C++ 标准；六层金字塔分层服务；Custom Entity + Object Enabler 平台经济学 |
| 3.2 | **CATIA CAA RADE** | 严格组件 + 接口模型；编译期强制 Authorized API 边界；Spec/Result/Update 声明式建模 |
| 3.3 | **Siemens NX (NX Open)** | 样本中较罕见的"四语言完全对等"SDK；Builder + Commit 范式；Synchronous Technology 直接编辑 + 关联性 |
| 3.4 | **Onshape** | 样本中较少见的 web-only 一等公民；Git 式数据模型（branch/merge）；FeatureScript 强类型 + 单位安全 |
| 3.5 | **MicroStation + iTwin** | "schema-first"工程哲学（ECSchema/BIS）；桌面 + 云双轨；iModel 联邦数字孪生 |
| 3.6 | **SolidWorks** | COM Automation 单层架构；IModelDocExtension 接口版本化；面向中端"上手快、深度有限" |
| 3.7 | **SketchUp Ruby** | 嵌入式脚本极简哲学代表；17+ 类高粒度 Observer；Face-based 简化几何换性能 |
| 3.8 | **FreeCAD** | 开源参数化 CAD 讨论中最常被引用的平台之一；App/Gui 严格 MVC 分离；Python 集成较深（FeaturePython）；22 年才到 1.0 |

### 3.5 五条历史演进趋势

```
趋势 1：分层化（Layering）
趋势 2：声明化（Declarativization）
       ⚠️ 在参数化 CAD 中明显，命令式平台不必然遵循
趋势 3：协作化（Collaborativization）
趋势 4：上云化（Cloudification）
       ⚠️ 云原生 + 本地部署双轨长期共存
趋势 5：智能化（Intelligencification）
```

### 3.6 6 条工程师良知

1. 审慎宣称 1.0
2. 尊重 ISV 投资
3. 透明沟通破坏性变更
4. 开放数据格式
5. 尊重用户的"不学习"权利
6. 避免锁定但允许深度集成

### 3.7 8 大陷阱清单

1. 低估几何内核工程量
2. 把 schema 系统当作"事后追加"
3. 忽视 TNP（拓扑命名问题）
4. 模仿成功平台的所有细节
5. 低估兼容承诺的成本
6. 过度设计未来
7. 忽视 ISV 痛点
8. 把云原生当作终点

## 六、常见问题（FAQ）

### Q1：证据等级 A/B/C 是什么？可以信吗？

A/B/C 是文档 1 引入的**写作辅助标签**——用于提示读者哪些论断有更扎实的样本支撑（A），哪些是作者归纳（B/C）。

⚠️ **A/B/C 不是客观评分**——它是作者自评。读者应：
- 把 A/B/C 当作"作者对自己论断信心的提示"
- 不把它当作"已经被同行评议过的可信度认证"
- 重要决策时，对所有 B/C 等级的论断保持质疑

### Q2：本系列覆盖了哪些 CAD 平台？未覆盖的有哪些？

**已覆盖（8 个）**：
- 机械 CAD：CATIA、NX、SolidWorks、Onshape、FreeCAD（5 个）
- AEC + 设计师：AutoCAD、MicroStation+iTwin、SketchUp（3 个）

**未覆盖**的重要平台包括 PTC Creo、Autodesk Inventor、Solid Edge、Rhino + Grasshopper、BricsCAD、Revit、ArchiCAD 等。本系列若有续篇可补充。

### Q3：本系列的论断可以直接作为决策依据吗？

⚠️ **不建议直接作为决策依据**。本系列的所有论断都是**思考工具**——基于 8 样本平台的归纳，而非普遍规律。

具体决策应：
- 把本系列论断作为"思考清单"
- 结合自身组织的客户基础、商业目标、生态阶段
- 对所有 B/C 证据等级的论断保持质疑
- 中国市场判断应核对市场调研机构数据

### Q4：本系列覆盖了哪些时间范围？

本系列反映 **2024-2026 初期** 的 CAD 业界状态。云原生、AI 整合、开源 CAD 等领域演进迅速，部分判断在未来 3-5 年可能需要更新。

## 七、写作方法说明

本系列采用以下方法保证可读性与可审计性：

- **来源标注**：3.x 系列报告中的事实陈述带 `<sup>[类别 N]</sup>` 来源标注
- **推论 / 评论分离**：用 `> **[推论]**` `> **[评论]**` 块引用区分作者判断
- **勘误公开**：⚠️ 标注对常见社区资料的事实修正
- **章节级交叉引用**：文档 1 与文档 2 中的关键论断引用到具体 3.x 章节
- **章节开头共性边界**：每个理论章节开头声明适用范围
- **证据等级标识**：A/B/C 写作辅助标签
- **行业观察附录定位**：中国市场内容明确不构成市场研究结论
- **样本范围内表述**：避免"所有 CAD 平台"等全称命题，使用"在样本中""在本文样本覆盖的平台中"等明确范围
